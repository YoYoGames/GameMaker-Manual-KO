명령줄 매개변수

게임을 만들 때 실행 파일을 테스트하여 최종 컴파일이 정확하고 예상치 못한 오류나 차이가 없는지 확인하는 것이 중요합니다. 그러나 실행 파일을 생성한다는 것은 디버그 콘솔이나 컴파일러 창에 액세스할 수 없기 때문에 발견할 수 있는 문제를 파악하고 디버그하는 것이 어려울 수 있음을 의미합니다. 그러나 이에 대한 수정 사항이 있습니다. 즉, 게임의 실행 가능한 버전이 실행될 때 명령줄 매개변수 를 사용하여 특정 속성을 설정하여 출력 로그 파일을 생성하고 다른 특별한 상황을 테스트할 수 있도록 하는 것입니다.

참고 이것은 표준 Windows 대상 모듈 에서만 사용할 수 있습니다.

이 작업을 수행하려면 먼저 게임의 Windows 실행 파일을 만들고 테스트 시스템에 설치해야 합니다. 완료되면 바탕 화면에 게임에 대한 바로 가기 를 만들어야 합니다. 여기에 전달할 매개변수를 추가합니다. *.exe (바로가기를 만들려면 오른쪽 클릭 게임 실행 파일을 선택한 다음 보내기 > 바탕 화면(바로 가기 만들기) 을 선택합니다.

바로 가기를 만든 후에는 대상 경로에서 다음 매개변수 중 하나를 사용할 수 있습니다.

-noaudio: 게임의 모든 오디오를 끕니다.

-inawindow: 전체 화면 모드로 실행되도록 설정된 경우에도 게임을 창에서 강제로 시작합니다.

-output <filename>: 코드 베이스에 있는 디버그 메시지를 포함하여 콘솔 출력을 파일 이름으로 보냅니다.

-debugoutput <filename>: 사용자 정의 디버그 메시지를 제외하고 버그 보고를 위한 러너의 추가 정보를 포함하여 콘솔 출력을 파일 이름으로 보냅니다.

-software: 게임 그래픽을 렌더링하기 위해 하드웨어 대신 소프트웨어 정점 처리를 사용하도록 게임을 강제합니다. 이것은 온보드 Intel GFX 칩셋을 사용하거나 구형 시스템에서 디스플레이 문제를 표시하는 PC에 특히 유용합니다.

-intel: 사용 중인 GPU에 상관없이 잘못된 Intel 드라이버에 대한 수정을 켭니다. 이러한 불량 드라이버에 대한 이 해결 방법은 속도를 늦추고 게임에 눈에 띄는 성능 저하를 제공하므로 수정이 필요하지 않은 경우 사용하지 마십시오.

-vanillaGFX: 이 옵션을 사용하면 드라이버 제조업체에 대한 모든 검사가 꺼지고 게임을 "있는 그대로" 테스트하는 데 매우 유용할 수 있습니다.

주목할 가치가 있습니다. <filename> 안에 -output 그리고 -debugoutput 파일이 각 쓰기에서 열리고 닫히므로 원하는 경우 동일한 파일일 수 있습니다.

이제 게임 실행 파일에 대한 바로 가기를 만든 곳으로 이동한 다음 마우스 오른쪽 버튼을 클릭하고 속성을 선택해야 합니다.

그러면 위와 같은 창이 열립니다. 여기에서 명령줄을 대상 경로에 추가할 수 있습니다. 주어진 경로의 끝으로 스크롤하고 공백을 추가한 다음 매개변수 문자열을 추가하기만 하면 됩니다. 예를 들어 디버그 파일 출력을 생성하려면 다음과 같이 됩니다.

D:\Users\Me\GameMaker\CustomLoadBar-Default-1.0.0.0\CustomLoadBar.exe -debugoutput debug.txt

이제 "라는 파일이 생성됩니다.debug.txt " 실행 파일과 같은 위치(바로 가기 위치가 아니라 게임이 설치된 위치)에 완성된 게임의 디버그 및 컴파일러 정보가 있습니다. "확인"을 클릭한 다음 바로 가기를 사용하여 게임을 실행하여 이러한 항목을 사용하도록 합니다. 매개변수 및 디버깅에 도움이 됩니다.

게임을 디버그하는 방법에 대한 추가 정보는 매뉴얼의 디버깅 섹션을 참조하십시오.

명령줄을 사용한 A/B 테스트

또한 YoYo 러너는 다음과 같은 특수 명령줄 매개변수의 이점을 누릴 수 있습니다.

-game <filename>: 주어진 게임 파일을 로드합니다.

왜 이것을 사용할까요? 글쎄, 일반적으로 그렇게 하지 않을 것이지만, 한 번에 둘 이상의 게임 버전을 실행하고 싶다면 이것을 사용하여 A/B 비교 테스트를 훨씬 쉽게 만들 수 있습니다. 이 방법을 사용하면 다른 사람과 테스트에 집중하거나 디자이너로서 나란히 테스트를 수행하여 사물에 대해 더 나은 느낌을 얻을 수 있습니다.

이를 설정하려면 사전에 몇 가지 사항을 알아야 합니다. 시작하려면 Runner 자체에 대한 경로가 필요합니다. 이것은 Windows OS의 Program Data 폴더에서 찾을 수 있습니다(기본적으로 숨겨져 있을 수 있으므로 보기 속성을 변경해야 볼 수 있음). 일반적인 형식은 다음과 같습니다.

C:\ProgramData\GameMaker\Cache\runtimes\<runtime_version>\windows\Runner.exe

runtime_version 폴더는 현재 설치한 런타임에 따라 다릅니다. 당신은 또한 경로가 필요합니다 <yourgame>.win 이것이 우리가 주자를 가리킬 것이기 때문입니다. 이것을 찾는 가장 쉬운 방법은 Temp 폴더(일반적으로 %localappdata%\Temp\GameMakerStudio\<game_name>\ )를 검색하고 *.win, 사용 가능한 모든 항목이 표시되며 전체 경로를 가져오는 데 필요한 항목을 간단히 선택할 수 있습니다.

이제 실행 파일에 대해 수행한 것처럼 러너에 대한 바로 가기를 만든 다음 필요에 따라 매개변수를 변경할 수 있습니다. -측면 테스트. 대신 우리는 이것을 처리하기 위해 몇 개의 작은 "배치" 파일을 만들 것입니다.

이렇게 하려면 먼저 바탕 화면에 새 텍스트 문서를 만들고 메모장(또는 다른 텍스트 편집기)에서 엽니다. 다음을 추가합니다.

START C:\ProgramData\GameMaker\Cache\runtimes\<runtime_version>\windows\Runner.exe -game <LOCAL TEMP PATH>\<GAMENAME>.win

게임을 가리키도록 임시 경로를 설정해야 합니다. *.win 파일을 만든 다음 "GM_Test_1.bat "를 바탕 화면에 저장합니다(메모장은 자동으로 *.txt 저장 대화 상자 옵션에서 "모든 파일"을 선택하지 않는 한 파일). 최종 명령이 어떻게 표시되어야 하는지에 대한 예는 다음과 같습니다.

START C:\ProgramData\GameMaker\Cache\runtimes\runtime-9.9.1.199\windows\Runner.exe -game D:\Users\Mark\AppData\Local\Temp\GameMaker\AlphaDog_Reboot\AlphaDog_Reboot.win

이제 다른 텍스트 파일을 만들고 같은 줄을 추가합니다. 이번에는 나란히 테스트하려는 게임의 두 번째 버전 경로를 가리키고 이것을 "GM_Test_2.bat ". 두 번 클릭할 수 있습니다. 둘 중 하나를 사용하여 게임의 다른 버전을 불러오고 배치 파일로 저장하면 쉽게 다시 열어 테스트 중인 게임을 편집하고 변경할 수 있습니다.
