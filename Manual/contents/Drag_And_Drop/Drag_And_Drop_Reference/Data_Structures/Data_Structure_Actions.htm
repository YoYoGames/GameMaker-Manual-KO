데이터 구조 작업 라이브러리

게임에서는 종종 정확하고 질서 있는 방식으로 정보를 저장해야 합니다. 예를 들어, 사람이 가지고 다니는 항목 목록을 저장해야 하거나 여전히 방문해야 하는 장소의 그리드를 저장해야 할 수 있습니다. 이제 이를 위해 여러 변수를 사용할 수 있지만 데이터 정렬이나 특정 항목 검색과 같은 더 복잡한 작업을 수행하려면 실행 속도가 느리고 실행하기 어려울 수 있는 많은 양의 작업 블록 코드를 작성해야 합니다. 디버그하고 작성하기가 정말 어렵습니다!

이를 해결하기 위해 GameMaker 에는 전문 기능을 통해 액세스할 수 있는 여러 내장 데이터 구조가 있습니다. GML Visual을 통해 사용할 수 있는 데이터 구조에는 4가지 유형이 있습니다( GML 을 사용하는 경우 6가지 있음). 저장하려는 정보 유형과 나중에 조작하려는 방법에 따라 각각 장단점이 있습니다. - 네 가지 유형은 다음과 같습니다.

기울기 목록은 정보가 추가될 때 순차적으로 정보를 저장하는 데이터 구조입니다. 목록 인덱스는 0에서 시작하여 목록 끝에 추가되는 각 항목에 대해 올라가지만 마지막 위치가 아닌 다른 위치에 데이터를 삽입할 수도 있습니다. 스택 데이터 구조(아래 설명)와 달리 목록의 모든 인덱스에서 언제든지 데이터에 액세스할 수 있습니다.
지도 맵 데이터 구조는 키 와 값 쌍 을 저장할 수 있으므로 매우 유용합니다. 예를 들어 게임의 캐릭터는 다양한 항목(키)을 가질 수 있고 각 개별 항목에 대해 여러 가지(값)를 가질 수 있습니다. 예를 들어 RPG에서 10개의 건강 물약, 5개의 마나 물약 및 100골드. 지도는 이러한 쌍을 모두 한 곳에서 유지하고 지도에 쌍을 추가하고 몇 가지 간단한 기능을 사용하여 특정 키에 해당하는 값을 검색할 수 있습니다. 맵은 (인식 가능한) 어떤 방식으로도 정렬되지 않습니다. 즉, 특정 키를 찾으려면 전체를 반복해야 할 수 있으며(매우 느림) 동일한 두 개의 키를 보유할 방법도 없습니다. 하나의 키에 두 개의 값을 할당할 수 있습니다.
그리드 그리드는 너비와 높이를 정의하는 2차원 값 배열입니다. 이것은 그리드 내의 그리드 "셀"의 총 수(너비와 높이 곱함)를 설정하고 이러한 셀은 허용된 데이터 유형이 될 수 있는 다른 데이터 값을 보유하는 데 사용됩니다. 이 유형의 구조를 사용하면 셀의 x(열) 및 y(행) 위치를 지정하여 그리드의 셀 값을 설정하고 검색할 수 있습니다(행과 열 모두 0에서 시작하여 그리드 너비 - 1 및 그리드 높이 -1). 그리드 데이터 구조에 접근할 때 항상 셀 위치에 정수 값을 사용하려고 시도해야 하며 정수가 아닌 모든 인덱스는 사용하지 않는 경우 GameMaker Studio 2에 의해 바닥이 됩니다. 이것이 필요하지 않은 경우 확인하려는 인덱스를 전달하기 전에 미리 반올림을 수행해야 합니다.
스택 스택 데이터 구조는 소위 LIFO(후입 선출) 구조입니다. 스택에 값을 푸시하고 스택에서 값을 다시 꺼내 제거할 수 있으며 가장 최근에 스택에 푸시된 값이 스택에서 가장 먼저 꺼집니다(단지 동전 스택을 생각해 보세요. 추가한 동전은 아래의 나머지 동전에 도달하기 전에 먼저 다시 제거해야 합니다. 스택은 처리해야 할 인터럽트가 있거나 재귀 함수가 있거나 게임을 위한 기초적인 AI를 구성할 때에도 자주 사용됩니다.

기본적으로 모든 데이터 구조는 동일한 방식으로 작동합니다. 데이터 구조를 만들고 해당 인덱스 값을 변수에 저장합니다. 그런 다음 이 인덱스를 사용하여 작업을 수행할 수 있는 모든 추가 함수 호출에서 데이터 구조를 참조합니다. 마지막으로 작업이 완료되면 데이터 구조를 다시 파괴하여 메모리에서 제거합니다. 필요한 만큼의 구조를 동시에 사용할 수 있으며 모든 구조는 문자열과 실제 값을 모두 저장할 수 있습니다.

참고: 모든 동적 리소스와 마찬가지로 데이터 구조는 메모리를 차지하므로 더 이상 필요하지 않을 때 항상 파괴 되어 속도가 느려지고 결국 게임이 중단되는 메모리 누수를 방지해야 합니다.
