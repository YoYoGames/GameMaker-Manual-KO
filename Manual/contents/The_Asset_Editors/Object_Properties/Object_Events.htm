개체 이벤트

그렇다면 객체 이벤트 는 무엇입니까? 기본적으로 이들은 게임 루프에서 사용자가 프로그래밍한 내용을 기반으로 일이 발생하는 신중한 순간입니다. GameMaker 는 이러한 이벤트의 주기로 작동합니다. 방이 시작되는 순간부터 완료되는 순간까지 모든 단계 에서 일련의 이벤트가 실행되거나 확인되는 게임 루프가 실행되며 코드 또는 GML 시각적 작업을 배치하도록 선택할 수 있습니다. 이러한 이벤트에 응답하는 개체에서(단계는 게임 시간의 한 순간이며 실내 속도 설정에 의해 제어되며 프레임 이라고도 할 수 있음).

이벤트 및 코드가 있는 일반적인 개체 설정을 살펴보겠습니다.

예제 객체에서 볼 수 있듯이 응답해야 하는 많은 이벤트가 나열되어 있지만 처음에는 객체를 생성할 때 이 목록이 비어 있으며 필요한 이벤트와 해당 객체의 인스턴스가 이벤트가 트리거됩니다. 개체에 이벤트를 추가하려면 이벤트 목록 하단에 있는 이벤트 추가 버튼을 누르면 다음 창이 나타납니다.

이것은 개체가 응답할 수 있는 모든 기본 이벤트 및 이벤트 범주의 목록이며, 각 범주 내에는 동작을 더욱 세분화하기 위한 하위 이벤트가 있습니다. 예를 들어 키 누름 을 클릭하면 이벤트 범주에서 개체가 응답해야 하는 키를 선택할 수 있는 이벤트가 포함된 추가 창이 나타납니다.

이벤트를 선택하면 이벤트의 편집기 창이 이벤트에 연결되어 오른쪽에 열립니다.

이제 GML 코드(또는 블록)를 편집하여 개체에 해당 이벤트에 대한 특정 동작 또는 반응을 부여할 수 있습니다.

마우스 오른쪽 버튼을 클릭할 수 있습니다 개체에 추가된 모든 이벤트에서 다음 메뉴 옵션을 가져옵니다.

이러한 옵션은 다음과 같습니다.

  • 이벤트 추가 - 이벤트 목록에서 새 이벤트를 추가합니다.
  • 이벤트 잘라내기 / 복사 / 붙여넣기 - 선택한 이벤트를 잘라내거나 복사하거나 붙여넣습니다. 표준 키보드 단축키도 사용할 수 있습니다. / + X, / + C, 그리고 / + V.
  • 이벤트 복제 - 선택한 이벤트를 복제합니다. 이렇게 하면 선택한 이벤트의 내용을 가져와서 지정한 다른 이벤트에 복제합니다.
  • 이벤트 변경 - 이벤트 를 변경합니다. 이렇게 하면 다른 이벤트 범주를 선택하라는 메시지가 표시되고 일단 선택하면 현재 이벤트의 내용이 새 이벤트로 변경됩니다(프로세스에서 원래 이벤트 제거).
  • GML Visual/Code로 변환 - 코드를 사용하거나 GML Visual 작업을 사용하여 프로젝트를 생성할 때 선택한 방법을 사용하여 개체 이벤트가 생성됩니다. 이 옵션을 사용하면 프로젝트 유형에 관계없이 하나에서 다른 것으로 변환할 수 있습니다.
  • 이벤트 삭제 - 이벤트 를 삭제합니다.

이벤트를 제거할 때 다음을 사용할 수 있습니다. + 여러 이벤트를 선택한 다음 모두 함께 삭제합니다. 생성한 각 개체에는 개체 편집기 에서 추가되는 고유한 이벤트 목록이 있습니다. 이러한 이벤트는 두 가지 범주로 나뉩니다.

  • 모든 단일 게임 단계를 실행하는
  • 인스턴스가 방 가장자리에 도달하거나 키보드 또는 마우스를 누르는 것과 같은 게임 이벤트에 의해 "트리거되는" 것입니다.

전체 이벤트 목록은 다음과 같습니다.

만들다

이 이벤트는 객체의 인스턴스가 처음 생성될 때 발생하며, 방에 들어갈 때 Room Editor를 통해 방에 배치된 인스턴스 내에서 가장 먼저 발생하는 이벤트입니다. 이것은 이 이벤트가 변수 s 를 초기화하고, 타임라인 을 시작하고, 경로 를 설정하고... 일반적으로 한 번만 또는 인스턴스가 룸에서 처음 생성될 때만 수행되어야 하는 다른 모든 작업을 수행하기에 이상적인 장소임을 의미합니다. 개체에 개체 편집기 또는 룸 편집기에 추가된 개체 변수 또는 인스턴스 변수 가 있는 경우 이러한 변수가 먼저 초기화된 다음 이벤트 생성 이 실행됩니다.

룸 편집기의 인스턴스 생성 코드 에서 이벤트 생성에서 설정한 모든 것을 수정할 수 있습니다. 이는 인스턴스에 대한 생성 이벤트 직후 실행되고 인스턴스 변수 를 생성하거나 다음과 같이 추가된 변수를 재정의하는 데 사용할 수 있습니다. 개체 변수 또는 실제 만들기 이벤트에서.

파괴하다

이 이벤트는 인스턴스가 소멸될 때 실행되는 이벤트입니다. 개체에 동작을 추가할 때 종종 간과되지만, 예를 들어 적이 죽었을 때 폭발 또는 입자 효과를 생성하거나 방의 다른 부분에서 개체의 새 인스턴스를 다시 생성하는 경우와 같이 매우 유용할 수 있습니다. 점수에 점수를 추가하는 경우에도.

정리

이 이벤트는 방에서 개체의 인스턴스를 제거하는 모든 이벤트 후에 호출됩니다. 따라서 다음과 같은 경우 트리거됩니다.

  • 인스턴스가 파괴됨
  • 방 끝
  • 게임이 종료됩니다

게임에 있을 수 있는 동적 리소스(표면, 데이터 구조 등...)를 "정리"하거나 인스턴스가 제거될 때 한 번 수행해야 하는 작업을 수행하는 데 사용하도록 설계되었습니다. 어떤 식 으로든 게임.

이 이벤트는 이벤트를 트리거한 이벤트 직후 에 호출되지만 현재 이벤트가 끝날 때까지 인스턴스는 실제로 게임에서 제거되지 않습니다. 예를 들어 전화를 걸면 instance_destroy() Step Event에서 Destroy Event가 호출되고 Clean Up Event가 호출 되고 나머지 Step Event 실행이 완료됩니다. 즉, 에 대한 호출 후 코드가 instance_destroy() 코드에 필요한 데이터 구조 또는 기타 리소스를 정리한 경우 여전히 실행되고 잠재적인 오류 원인이 되므로 이 이벤트를 사용할 때 주의해야 합니다.

경보

알람 범주는 인스턴스에서 설정할 수 있는 가능한 알람 각각에 대해 하나씩, 12개의 이벤트로 분할됩니다. 따라서 Add Alarm 범주를 클릭하면 다음 창이 표시됩니다.

여기에서 생성하려는 알람을 선택하고 완료되면 이벤트 창에 알람이 추가되어 평소처럼 코드를 추가할 수 있음을 알 수 있습니다. 근데 알람이 뭔가요? 음, 알람이 이전에 설정되지 않은 한 아무 일도 하지 않는 특별한 이벤트이며, 알람에 추가한 작업이나 코드를 실행하기 전에 해당 알람이 0으로 카운트다운될 때까지 대기합니다.

알람이 0에 도달하고 코드를 실행하면 -1까지 카운트다운되어 다시 설정될 때까지 유지됩니다(즉, 알람 값을 확인하여 알람 값이 -1보다 큰지 확인할 수 있습니다. 실행 중인지 여부). 그래서, 당신이 설정 alarm[0] 개체의 생성 이벤트에서 30으로 설정하면 GameMaker 가 30개의 게임 단계를 카운트다운한 다음 alarm[0] 이벤트. 알람을 0으로 설정하면 이벤트가 트리거되므로 알람 코드가 실행 되지 않지만 알람은 즉시 -1로 설정되므로 코드를 건너뜁니다. 바로 다음 단계를 실행하기 위해 알람이 필요한 경우 1로 설정해야 합니다.

이것은 정확한 순간에 움직이는 것을 설정할 수 있기 때문에 매우 유용할 수 있으며 자체 이벤트에서 알람을 설정하는 것을 막을 수 있는 방법이 없기 때문에 반복하도록 할 수도 있습니다. 몬스터가 있고 3초마다 우회전하기를 원한다고 상상해보십시오... 글쎄, 생성 이벤트에서 알람을 룸 속도 * 3으로 설정합니다(룸 속도가 30이면 초당 30보이므로 그것에 3을 곱하면 3초가 됩니다!) 그런 다음 알람 이벤트에서 방향을 설정하는 코드 또는 작업과 알람을 다시 실내 속도 * 3으로 설정하는 작업(또는 코드)이 있습니다. 이런 식으로 특정 간격에서만 발생하는 간단한 게임 루프를 설정할 수 있습니다.

동작이나 코드가 없는 알람은 카운트다운 되지 않습니다. 그러나 주석만 있고 코드나 작업이 없는 경우에도 알람은 계속 카운트다운되며 평소와 같이 설정하고 확인할 수 있습니다.

단계

GameMaker 는 초당 이러한 단계가 몇 개인지 정의하는 실내 속도로 시간을 단계 로 나눕니다(단계는 프레임 이라고도 함). 단일 단계는 기본적으로 게임이 실행되는 동안 필요에 따라 모든 이벤트를 확인하고 트리거하면서 지속적으로 실행되는 루프입니다. 따라서 상상할 수 있듯이 단계 이벤트 는 인스턴스가 실행되는 동안 게임의 모든 단계에서 확인되는 이벤트입니다. 존재합니다.

단계 이벤트는 실제로 아래에 설명된 세 가지 하위 이벤트로 구성됩니다.

대부분의 경우 표준 단계 이벤트를 사용하는 것이 좋지만 때로는 코드가 실행되는 시간과 실행 시간을 좀 더 세밀하게 제어하여 시작 및 종료 단계 이벤트가 제공됩니다. 세 가지 모두 모든 단계를 확인하지만 GameMaker 엔진에 대한 향후 업데이트가 다른 이벤트를 변경하더라도 순서는 변경되지 않습니다. 즉, 이것이 항상 다른 것보다 먼저 발생하도록 하는 신뢰할 수 있는 유일한 방법입니다.

단계 이벤트는 무엇에 사용할 수 있습니까? 글쎄요, 지속적으로 실행해야 하는 작업이나 코드에 사용할 수 있습니다. 예를 들어, 한 개체가 다른 개체를 따라야 하는 경우 여기에서 뒤에서 부드럽게 움직이도록 우리가 따르고 있는 개체에 대한 모션 방향을 조정할 수 있습니다. 하지만 이 이벤트에 주의하고 개체의 단계 이벤트에 복잡한 동작을 많이 넣지 마십시오. 특히 게임실에 개체의 인스턴스를 많이 둘 계획이라면 게임 속도가 느려질 수 있습니다. 많은 것들이 알람에 배치되거나 항상 발생하기 보다는 일부 기타 이벤트를 사용하여 트리거되도록 설정할 수 있습니다.

충돌

분명히 게임을 만들 때 객체의 두 개(또는 그 이상) 인스턴스가 충돌한 시점을 아는 것이 매우 중요합니다. 이를 위해 Collision Event 가 있습니다. 이것은 개체에 배치한 다음 충돌을 확인해야 하는 다른 개체를 지정하는 이벤트입니다.

물리 기능이 켜져 있지 않으면 이러한 충돌은 두 개체의 마스크(마스크는 스프라이트 속성 내에서 정의되거나 개체 속성에서 독립적으로 할당될 수 있음)와 겹치는지 여부를 기반으로 계산됩니다. 충돌의 인스턴스 중 하나 또는 다른 인스턴스에 마스크가 할당되지 않은 경우(또는 스프라이트 마스크가 아무것도 설정되지 않은 경우), 무언가를 그리더라도 충돌이 감지되지 않습니다.

Physics 가 켜져 있으면 충돌은 충돌에 대한 반응과 마찬가지로 물리적 속성에서 개체에 대해 정의한 충돌 모양 ( Fixture ) 유형을 기반으로 합니다. 즉, 충돌을 처리하는 데 코드가 필요하지 않을 수 있지만 충돌을 감지하려면 이 이벤트 에 최소한 주석이 있어야 합니다.

마지막으로 모든 충돌은 충돌 이벤트가 트리거되기 전에 게임 단계당 한 번씩 계산되므로 충돌 이벤트가 실행될 때 모든 충돌이 이미 계산되어 미리 할당된 것입니다. 즉, 이 이벤트에서 인스턴스를 생성한 다음 충돌을 확인하려고 하면 게임 루프의 다음 반복 까지 충돌이 감지되거나 해결되지 않습니다.

건반, 키보드 프레스, 키보드 릴리스

플레이어가 게임의 다양한 측면을 제어하도록 하는 것은 매우 중요하며 이를 위해 GameMaker 는 세 가지 주요 키보드 범주에서 사용할 수 있는 매우 포괄적인 키보드 이벤트 목록을 제공합니다. 일반 키보드 범주의 경우 선택한 키를 누르고 있는 동안 매 단계마다 연속적으로 트리거되는 반면, Press and Release 범주 이벤트는 키를 처음에 누르거나 놓을 때 한 번만 트리거됩니다.

키보드 이벤트는 키가 사용될 때마다 방의 모든 활성 인스턴스에서 실제로 트리거되지만 해당 특정 키에 대해 정의된 이벤트가 있는 이벤트만 응답하고 모든 개체 및 인스턴스에서 여러 키보드 이벤트를 생성할 수 있습니다. 게임이 실행되는 동안 해당 개체의 모든 개체에 응답합니다.

개체에 키보드 이벤트를 추가하면 확인할 키를 지정할 수 있는 키보드 하위 이벤트 메뉴가 표시됩니다.

대부분은 매우 명확하지만 섹션을 간략히 살펴보겠습니다. 맨 위에 화살표 키가 있고 그 다음이 가장 많이 사용되는 수정자 키, 그 다음 키보드의 나머지 부분이 있습니다. 필요한 정확한 키 또는 ) 그리고 마지막으로 두 개의 매우 특별한 하위 이벤트인 No Key 및 Any Key. 이름에서 알 수 있듯이 이들은 키를 누르지 않았을 때 또는 아무 키나 누를 때를 확인하는 하위 이벤트입니다. 숫자 키패드의 키는 숫자 잠금 이 활성화된 경우에만 해당 이벤트를 생성한다는 점에 유의하십시오.

키보드의 Press 및 Release 이벤트는 연속적으로 트리거되는 대신 한 번만 트리거된다는 점을 제외하면 일반 키보드 이벤트와 거의 동일합니다. 키보드가 처음으로 키가 눌렸다고 등록하면 Keyboard Pressed 이벤트(일반 키보드 이벤트와 마찬가지로)가 발생하고 그 이후 처음으로 키가 눌린 것으로 감지되지 않는 경우 단일 키보드 가 트리거됩니다. 출시 이벤트.

쥐

마우스 카테고리는 게임에서 일어나는 일을 보다 정확하게 제어할 수 있도록 선택할 수 있는 일련의 이벤트로 구분됩니다. 여기에서 이러한 이벤트가 정확히 무엇인지 확인할 수 있습니다.

왼쪽, 오른쪽 그리고 중간 버튼 이벤트( 일반, 눌림 또는 해제 )는 모두 이벤트가 있는 인스턴스의 마스크에서 작동합니다. 이것이 의미하는 바는 마우스 이벤트가 있는 인스턴스의 충돌 마스크에 대해 버튼이 사용될 때 GameMaker 가 방에서 마우스의 위치를 확인한다는 것입니다. 인스턴스 경계 상자 에 "충돌"이 있는 경우 이벤트가 트리거되므로 이러한 이벤트가 있는 모든 인스턴스에 유효한 충돌 마스크가 있는 스프라이트가 있는지 또는 개체에 개체 속성에서 선택된 마스크 스프라이트가 있는지 확인하십시오. 이름에서 알 수 있듯이 이러한 이벤트는 선택한 마우스 버튼을 처음 누르거나 놓을 때 한 번만 트리거되거나 버튼이 유지되는 동안 각 단계마다 계속 트리거됩니다.

마우스 들어가기 및 나가기 이벤트는 작동하기 위해 인스턴스의 마스크에 너무 의존한다는 점에서 버튼 이벤트와 유사하지만, 이번에는 마우스가 인스턴스에 처음 "들어가"(터치)하거나 마우스가 " 나뭇잎"(만지기를 멈춤) 인스턴스. 그러나 이러한 이벤트는 연속적이지 않으며 마우스가 개체에 들어가거나 나갈 때마다 한 번만 트리거됩니다. 마우스를 제거하면 정상입니다.

마지막으로 Global Mouse 라고 하는 마우스 이벤트에 대한 또 다른 섹션이 있습니다. 이 하위 메뉴에서는 마우스가 마우스 위에 있지 않거나 가까이 있지 않은 경우에도 마우스 이벤트를 기록하기 위한 이벤트 선택을 찾을 수 있습니다. 이는 모든 인스턴스에 대해 생성되는 이벤트이며 지정된 이벤트에 대해 정의된 작업 또는 코드가 있으면 게임 룸 내에서 마우스의 위치에 관계없이 실행됩니다.

모바일 또는 터치 스크린 장치에서 마우스 왼쪽 버튼 터치 스크린의 손가락 탭과 마우스 오른쪽 버튼을 확인하는 데에도 사용할 수 있습니다. 화면을 두 번 탭하면 트리거됩니다(이 동작은 코드를 사용하여 변경할 수 있음).

제스처

이 이벤트는 사용자가 화면을 터치하거나(모바일에서) 마우스를 클릭하고 움직이면(다른 모든 플랫폼에서) 트리거되는 이벤트입니다. 이러한 이벤트는 일반 버전과 글로벌 버전이 있다는 점에서 마우스 이벤트와 유사합니다. 이러한 이벤트의 일반 버전은 스프라이트(또는 마스크)가 있는 인스턴스에서 터치가 발생하고 경계 상자 내에서 터치가 발생하는 경우에만 트리거됩니다. 그러나 이러한 이벤트의 전역 버전은 사용자가 화면의 아무 곳 이나 터치하면 트리거됩니다.

제스처 이벤트는 다음을 감지합니다.

  • 탭 - 사용자가 빠르게 클릭/터치하고 놓을 때
  • 끌기 - 사용자가 터치/클릭하고 유지한 다음 손가락/커서를 움직일 때
  • flicks - 사용자가 한 번의 움직임으로 터치/클릭을 이동하고 놓을 때
  • 핀치 - 사용자가 화면에 두 개의 손가락을 대고 손가락을 모으거나 벌릴 때
  • 회전 - 사용자가 화면에 두 손가락을 놓고 한 점을 중심으로 회전할 때

다른 이벤트에는 항상 "event_data " 지도에는 터치/클릭 위치 및 이동에 대한 데이터와 함께 여러 키/값 쌍이 포함됩니다. 사용 가능한 모든 하위 이벤트와 작동 방식에 대한 자세한 내용은 다음 섹션을 참조하세요.

그리다

이 이벤트 범주는 게임을 실행할 때 화면에 표시되는 내용을 제어하는 범주이며 다양한 개별 이벤트로 나뉩니다.

보시다시피 그리기 이벤트 범주에는 여러 가지 이벤트 유형이 있습니다. Draw Begin, Draw 및 Draw End 는 아마도 가장 많이 사용하게 될 "표준" 그리기 이벤트입니다. 기본적으로 기본 그리기 이벤트는 스프라이트가 있는지 여부에 관계없이 모든 인스턴스에 대해 항상 호출되지만 인스턴스를 보이지 않는 것으로 플래그 지정하면 이벤트가 트리거되지 않습니다(따라서 게임 논리가 있는 경우 이 점을 염두에 두십시오. 실행되지 않으므로 보이지 않는 개체의 그리기 이벤트에서). 메인 그리기 이벤트는 이벤트에 코드나 액션이 없을 때 GameMaker 기본이 인스턴스 스프라이트를 그리는 곳이기도 합니다(즉, 개체의 이벤트 목록에 추가하지 않은 경우). 기본 그리기는 인스턴스와 연결된 스프라이트를 사용하며 적용된 코드 또는 작업에 설정된 변환으로 이를 그립니다.

표준 그리기 이벤트는 Draw GUI 이벤트 이전 과 Pre Draw 와 Post Draw 이벤트 사이 에 그려집니다. 즉, 이 이벤트에서 그려지는 모든 것은 레이어에 관계없이 Draw GUI 이벤트 아래에 그려집니다(즉: 그리기 GUI 이벤트는 레이어 순서에 관계없이 항상 일반 그리기 이벤트에서 그려지는 모든 것 위에 그려집니다.

비동기
