개체 편집기

개체는 게임의 측면을 제어하고 특정 작업을 수행하는 데 사용하는 리소스입니다. 대부분의 경우 게임 룸에서 볼 수 있도록 연결된 스프라이트가 있지만 때로는 사용자 또는 타이밍 등과 관련된 작업을 수행하기 위해 "뒤에서" 컨트롤러로 사용됩니다... 행동이 주어질 수 있고 특정 이벤트에 반응할 수도 있고 서로에게 반응할 수도 있습니다. 게임에서 보는 대부분의 사물은 개체와 개체의 상호 작용을 기반으로 합니다. 이제 "기반"이라고 하는 이유는 실제로 개체를 게임 룸에 직접 배치하는 것이 아니라 기본적으로 개체 리소스의 복사본(또는 원하는 경우 복제)인 이러한 개체의 인스턴스 를 배치하기 때문입니다. 이것은 인스턴스와 객체 가 같은 것이 아니며 각각에 영향을 줄 수 있는 고유한 기능 집합을 가지고 있기 때문에 기억해야 하는 매우 중요한 사항입니다.

이를 명확히 하기 위해 GameMaker 로 만든 게임을 생각해 보십시오. 모든 캐릭터, 몬스터, 공, 벽 등은 자산 브라우저 에서 생성한 모든 개체입니다. 그런 다음 개체 를 Room Editor 의 방으로 끌어 기본 개체 템플릿에서 복사한 인스턴스 를 만듭니다. 그런 다음 방의 인스턴스는 편집기 자체에서 또는 코드를 통해 변경될 수 있으므로 크기를 조정하거나 색상을 지정하거나 "기본" 개체에서 수정된 기타 세부 사항을 가질 수 있습니다. 따라서 인스턴스 에 영향을 미치거나 변경하는 무언가에 대해 이야기할 때 방에 있는 개체의 특정 복사본에 영향을 미친다는 의미입니다. 해당 단일 인스턴스만 영향을 받고 나머지는 모두 영향을 받지 않습니다. 그러나 객체 에 영향을 미치거나 변경하는 것에 대해 이야기할 때 자산 브라우저에서 객체를 수정하고 그에 대해 수행하는 모든 작업이 해당 지점에서 생성된 모든 인스턴스에도 반영된다는 것을 의미합니다. 따라서 개체는 인스턴스의 템플릿이고 인스턴스는 게임을 만들기 위해 방에 배치하는 것입니다.

게임에서 개체를 만들려면 마우스 오른쪽 버튼을 클릭합니다. 자산 브라우저에서 개체 만들기 옵션을 클릭합니다. 그러면 이 페이지 상단에 다음 섹션이 포함된 창이 열립니다.

세부

여기에서 개체에 이름을 지정할 수 있습니다. 이 이름에는 공백이나 승인되지 않은 문자가 포함되어서는 안 되며(영숫자 문자와 밑줄 "_"만 허용됨) 짧지만 기억하기 쉬워야 나중에 리소스가 많을 때 리소스를 쉽게 식별할 수 있습니다. 예를 들어 많은 사람들이 접두사 또는 접미사를 사용하여 "obj_Ship " 또는 "Ship_obj ".

다음으로 개체에 스프라이트가 필요하면 해당 스프라이트를 선택할 수 있습니다. 마우스 왼쪽 버튼으로 클릭 스프라이트 상자에서 자산 탐색기 가 자산 브라우저에서 사용 가능한 모든 스프라이트 목록과 함께 나타납니다.

그런 다음 개체에 사용할 항목을 선택할 수 있습니다. 개체에 할당된 스프라이트로 스프라이트 편집 버튼을 클릭할 수 있습니다. 스프라이트 속성 을 변경하거나 - 개체에 대한 스프라이트가 없지만 생성하려는 경우 스프라이트 추가 버튼을 클릭할 수 있습니다. 새 스프라이트 자산을 만들고 할당합니다. 이미지 편집 버튼을 클릭하여 할당된 스프라이트 자산 이미지를 편집할 수도 있습니다. 이미지 편집기 를 엽니다.

충돌 마스크

스프라이트 할당 영역 아래에는 특정 콜리전 마스크 를 선택하기 위한 버튼이 있습니다. 기본적으로 개체는 할당된 스프라이트를 기반으로 충돌 속성을 생성하지만, 충돌이 다른 스프라이트의 다른 모양을 기반으로 하는 경우가 있습니다. 예를 들어, 메인 스프라이트에 직사각형 충돌 마스크가 있는 하향식 게임이 있을 수 있지만 충돌을 더 부드럽게 만들기 위해서는 원형 마스크가 더 나을 수 있습니다. 이 경우 Collision Mask 버튼을 클릭하고 다른 스프라이트 리소스를 선택할 수 있으므로 이제 개체의 인스턴스가 할당된 스프라이트를 그리지 만 기본 충돌 은 마스크 스프라이트에서 분리됩니다. 스프라이트 편집기 페이지에서 충돌 마스크에 대한 자세한 정보를 찾을 수 있습니다.

속성

여기에 개체에 대한 다음 옵션이 제공됩니다.

  • Visible: Visible은 룸이 시작될 때 이 객체의 인스턴스를 볼 수 있는지 여부를 나타냅니다. 일반적으로 대부분의 인스턴스는 볼 수 있지만 때로는 보이지 않는 인스턴스가 있는 것이 유용합니다. 예를 들어 이동하는 몬스터를 제어하거나 값을 추적하고 특정 시간이 지정된 작업을 수행하기 위해 웨이포인트로 사용할 수 있습니다. 보이지 않는 객체는 여전히 이벤트에 반응하며, 스프라이트나 마스크가 할당되어 있는 경우 다른 인스턴스가 충돌할 경우에도 반응합니다... 그냥 볼 수 없고 이벤트 그리기 를 수행하지 않습니다. 기본적으로 보이는 플래그가 선택되어 있습니다.
  • 단색: 인스턴스를 단색으로 플래그를 지정하면 GameMaker 에 충돌 이벤트 를 트리거하기 전에 충돌 해결을 시도해야 한다고 말하는 것입니다. 이것은 매우 기본적인 기능이며 실제로는 충돌 이벤트 자체에 있는 코드 또는 GML Visual을 수행하기 전에 충돌이 감지되기 전의 위치로 인스턴스를 다시 이동하는 것뿐입니다. 제한된 사용.
  • Persistent: 영속 객체는 방이 변경될 때 사라지지 않고 "지속"되어 새 방으로 옮겨지는 것입니다. 파괴 작업이나 코드에서 명시적으로 파괴할 때만 사라집니다. 즉, 한 방에 영구 개체를 배치하면 모든 후속 방에서 계속 사용할 수 있으며 개체 속성의 이벤트에 할당된 모든 작업을 계속 수행 합니다. 단, 이벤트 생성 은 새 방으로 변경할 때가 아니라 처음에 생성됩니다. 이것은 예를 들어 방에서 방으로 이동하는 주인공이 있고 인스턴스 내에서 변수를 동일하게 유지하려는 경우에 편리합니다. 실수로 프로젝트에 오류가 발생합니다.

지속성 및 룸 레이어

방에서 지속성이 활성화된 인스턴스를 생성하면 레이어 또는 깊이가 할당됩니다(객체 인스턴스 생성에 사용한 기능에 따라 다름). 룸이 변경되고 다음 룸에 할당된 것과 동일한 이름이나 깊이를 가진 레이어가 없으면 룸 전체에 지속되는 인스턴스에 대해 새 레이어가 생성됩니다. 인스턴스를 생성할 때 명명된 레이어를 지정했다면 새 레이어의 이름도 원래 레이어와 동일하게 지정되고, 인스턴스에 깊이를 지정했다면 새 레이어의 이름은 "_layer_XXX ", 어디 "XXX "는 레이어에 고유한 이름을 지정하는 데 사용되는 16진수 값입니다. 이러한 레이어는 Room End 의 방에서 제거됩니다.

레이어 및 영구 인스턴스에 대해 주의해야 할 또 다른 사항은 생성 시 명명된 레이어를 인스턴스에 할당하고 다음 룸에 동일한 이름을 가진 다른 레이어가 있는 경우 지속형 인스턴스가 동일한 레이어에 할당된다는 것입니다. 레이어의 깊이에 관계없이 이름.

마지막으로, 지속 인스턴스가 인스턴스가 생성된 것과 동일한 깊이의 레이어가 있는 방으로 이동하면 이 레이어에 할당 되지 않고 대신 동일한 깊이에 새 레이어가 생성됩니다(이름 지정 규칙에 따라 위에서 설명).

지속성과 이벤트

이전에 언급했듯이 영구 개체는 Create Event 를 한 번만 실행하지만 Game Start, Game End, Room Start 및 Room End 이벤트도 적절한 시간에 트리거됩니다. 그러나 게임을 다시 시작하면(예: game_restart() 기능) 모든 영구 개체는 제거되고 게임에서 다시 생성될 때만 존재합니다.

또한 영구 개체를 비활성화 하면 Room End 이벤트가 트리거되기 전에 다시 활성화되지 않는 한 더 이상 한 방에서 다른 방으로 전달되지 않습니다. 재활성화 에는 전체 게임 단계가 필요 하므로 실제 Room End 이벤트를 사용하는 것은 작동하지 않습니다. 게임에 개체를 재활성화하는 전체 단계를 제공하지 않기 때문입니다. 따라서 사용할 때 이 점을 염두에 두어야 합니다. 영구 객체와 함께 비활성화하고 그에 따라 코드를 계획하십시오.

  • Uses Physics (물리 사용): Uses Physics(물리 사용) 상자를 선택하면 이 개체가 물리 세계의 일부여야 한다고 GameMaker 에 알리는 것입니다. 룸 에디터. 이 항목을 선택하면 이 개체의 모든 인스턴스에 대한 물리적 속성을 정의할 수 있는 추가 연결된 창이 열립니다. 이에 대한 더 자세한 정보는 Physics Objects 섹션에서 찾을 수 있습니다.
  • Managed: 롤백 멀티플레이어를 사용할 때 이 개체를 관리해야 하는지 여부입니다. 관리 대상 개체의 상태를 롤백 하고 예측을 실행할 수 있습니다. 관리 객체 를 참조하십시오.
