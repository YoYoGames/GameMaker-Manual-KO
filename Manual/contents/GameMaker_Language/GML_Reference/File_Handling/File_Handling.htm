파일 처리

게임을 만드는 동안 어느 시점에서 정보를 외부에 저장하고 이 정보를 검색할 수 있어야 합니다. 예를 들어 로그인 세부 정보, 최고 점수 또는 레벨을 통한 플레이어의 진행 상황을 저장할 수 있습니다. 이유가 무엇이든 GameMaker 로 작업할 때 사용할 수 있는 다양한 파일 기능을 사용하는 방법을 알아야 합니다.

GameMaker 는 기본적으로 샌드 박싱되어 있다는 것을 깨닫는 것이 매우 중요합니다. 즉, 사용 가능한 파일 시스템 기능을 사용할 때 많은 제한과 사소한 복잡성이 있으며 파일 시스템이 다음에 따라 약간 다르게 작동한다는 점도 주목할 가치가 있습니다. 타겟팅하는 플랫폼입니다. 이에 대한 자세한 내용은 다음 섹션을 참조하십시오.

  • 파일 시스템

선택한 플랫폼의 게임 옵션에서 데스크탑 대상(Windows, macOS 및 Ubuntu(Linux))에서 샌드박싱을 끌 수 있습니다. 즉, OS 권한의 제한 내에서 어디에서나 저장하고 로드할 수 있습니다.

Steam에서 게임에 전용 API를 사용할 수 있는 권한을 부여했다면 새로운 파일 기능 세트도 사용할 수 있습니다. 이는 이 설명서의 Steam API 섹션에 설명되어 있습니다.

마지막으로 파일 기능을 사용할 때 고려해야 하는 특정 대상 플랫폼 간에는 몇 가지 뚜렷한 차이점이 있다는 점에 유의해야 합니다. 이것들은:

  • iOS: iOS에서 폴더에서 포함된 파일을 로드하는 경우, 예: "Backgrounds/background1.png ", 폴더를 경로의 일부로 포함할 필요가 없으며 단순히 파일 이름만 제공하면 됩니다. 다른 모든 플랫폼에서는 파일 경로의 일부로 폴더를 제공해야 합니다.
  • macOS 및 Ubuntu(Linux): macOS 및 Ubuntu에서 파일 이름과 경로는 이러한 플랫폼에서 대소문자를 구분하므로 잠재적인 문제를 방지하기 위해 GameMaker 는 기본적으로 모든 이름/경로를 자동으로 소문자로 만듭니다.
  • HTML5: HTML5에서 로컬 저장소에 저장하고 로컬 저장소 에서 로드하는 것은 다른 기능 페이지에서 설명한 대로 작동하지만 서버에서 파일을 로드하는 경우에는 해당 기능만 사용해야 합니다.buffer_load_async() 동기 로딩은 대부분의 브라우저에서 더 이상 사용되지 않으며 결국 사용되지 않을 것입니다. 또한 브라우저 로컬 저장소에 디렉터리를 만들거나 삭제할 수 없습니다.

다양한 파일 기능에 대한 정보는 다음 페이지에서 찾을 수 있습니다.

  • 파일 시스템
  • 이니 파일
  • 텍스트 파일
  • 바이너리 파일
  • 파일 디렉토리

GameMaker 에는 파일과 데이터를 인코딩 하고 해싱 하기 위한 다양한 기능도 포함되어 있습니다. 여기에서 찾을 수 있습니다.

  • 인코딩 및 해싱

또한 다음 페이지에서 사용할 수 있는 게임 저장을 위한 몇 가지 기본 파일 기능이 있습니다(이 기능은 초보자가 기본 저장 시스템을 최대한 빨리 시작하고 실행할 수 있도록 설계되었지만 더 복잡한 프로젝트의 경우 위 섹션에 나열된 기능을 사용하여 자신의 저장 시스템):

  • game_load
  • game_load_buffer
  • game_save
  • game_save_buffer
