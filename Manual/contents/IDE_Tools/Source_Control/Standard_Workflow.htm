표준 워크플로

프로젝트에서 작업하면서 자연스럽게 스프라이트를 만들고, 스크립트를 편집하고, 타임라인을 삭제하는 등 프로젝트가 시간이 지남에 따라 발전함에 따라 필요한 모든 것을 하게 됩니다. 이것들은 모두 추적하고 문제가 발생하는 경우 되돌릴 수 있는 모든 작업입니다. 이것이 사고를 되돌릴 수 있도록 Source Control을 사용하는 주된 이유이며, 이것이 어떻게 작동하는지 이해할 수 있도록 처음부터 기본 워크플로를 신속하게 안내할 것입니다.

처음부터 새 GML 프로젝트를 만든 다음 기본 옵션 을 통해 소스 제어를 활성화합니다. 이렇게 하면 상단의 소스 제어 메뉴에 액세스할 수 있으므로 즉시 프로젝트 리포지토리 만들기 를 선택합니다. 방 리소스를 보면 기본방이 빨간색 느낌표가 붙은 것을 볼 수 있습니다., 마지막 상태에서 수정 되었음을 의미합니다. 프로젝트에 다른 리소스가 있는 경우 아래 이미지에 표시된 것과 같은 아이콘이 모두 있는 것을 볼 수 있습니다.

이제 빈 프로젝트를 그대로 커밋하거나 기본 상태로 가져오는 옵션이 있습니다. 지금은 첫 번째 커밋을 수행할 것이므로 Source Control > Commit Changes 를 클릭합니다. 이제 세 영역으로 분할된 새 창이 열립니다.

  • 단계적 변경 사항: 소스에 커밋될 변경 사항을 나타냅니다. 비어 있을 수 있지만 기본 옵션 변경 사항이 표시될 수 있습니다.
  • Unstaged Changes: 이것은 변경된 파일을 보여주지만 Git에 커밋하고 싶다고 말하지 않은 파일을 보여줍니다. 변경 사항은 "보류 중"입니다.
  • 커밋 메시지: 변경 사항을 설명하기 위해 추가할 수 있는 메모입니다.

지금은 모든 변경 사항을 커밋하기 위해 Stage All 버튼을 클릭하고 "First Commit!"과 같은 것을 입력합니다. 커밋 메시지에 입력하고 커밋 을 클릭합니다. 이제 모든 리소스에 녹색 체크 표시가 있어야 합니다., 수정 사항을 찾을 수 없음을 의미합니다.

이 창에서 작업할 때 다양한 Staged 및 Unstaged 메시지 앞에 기호가 붙습니다. 사용되는 기호는 작업 및 파일에 따라 다르며 다음이 사용됩니다.

  • [...] - 단계적 변경(이전에는 [A])
  • [+] - 새로 추가됨
  • [->] - 이름 변경/이동
  • [-] - 제거됨
  • [M] - 무단계 변경
  • [?] - 추적되지 않은 파일
  • [!] - 누락된 파일

이제 작업 프로세스를 시작할 수 있으며 위의 단락에서 수행한 것처럼 프로젝트를 빌드하고 변경 사항을 저장하고 커밋할 수 있습니다.

얼마 동안 작업한 후에는 언제든지 커밋한 내용을 확인하고 확인할 수 있으므로 커밋 기록 을 열고 싶을 것입니다. 기록 창을 열려면 소스 제어 > 기록 보기 로 이동하여 세 개의 창이 있는 다른 창을 엽니다.

위쪽 창은 커밋을 설명하고 중간 창은 선택한 커밋에 대한 설명을 설명하며 아래쪽 창은 이 커밋에서 수정된 파일을 설명합니다. Diff 도구 를 설정한 경우 이 창에서 파일을 두 번 클릭하면 diff 도구가 열리고 파일 간의 변경 사항을 볼 수 있습니다.
